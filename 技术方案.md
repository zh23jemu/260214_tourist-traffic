## 荔波县旅游客流分析与预测系统实施方案（可答辩原型）

  ### 一、文档内容总结

  1. 研究目标：基于多源数据（气象、节假日、票务/游客量、社媒等）分析荔波县客流影响因素，并建立预测模型，为管理决策提供
     支持。
  2. 总体框架：采用三层架构（UI/BLL/DL），覆盖采集、存储、清洗、分析、建模、可视化全流程。
  3. 技术方向：文档已明确 Python 生态（Pandas/NumPy/Scikit、Matplotlib/Seaborn/Plotly）、数据库（MySQL），前端建议 Vue
     + ECharts。
  4. 关键需求：

  - 功能：数据获取、存储、处理分析、可视化、预测结果展示。
  - 性能：查询/分析响应约 5 秒内，页面响应约 7 秒内。
  - 安全：数据访问控制、备份、敏感数据保护。

  5. 当前不足：模型选择与上线流程偏泛化，缺少“可直接开发”的数据标准、接口契约、模块边界和验收标准。

  ### 二、推荐开发路线（与你选择一致）

  采用 Python + FastAPI + MySQL + Vue3 + ECharts，目标是 20 周内交付“可答辩原型”，预测任务为“日级未来 7/30 天”。

  ### 三、决策完成版系统设计

  #### 1. 范围定义（一期）

  1. 纳入范围：

  - 日级历史数据管理（游客量、气象、节假日、事件标签）。
  - 影响因素分析（相关性、特征重要性）。
  - 预测（未来 7/30 天）与可视化看板。
  - 基础登录鉴权、操作日志、数据导入导出。

  2. 暂不纳入：

  - 小时级实时预测。
  - 分布式 HDFS/NoSQL（原型阶段先不用）。
  - 复杂多租户与高可用集群。

  #### 2. 技术架构

  1. 前端：Vue3 + Vite + ECharts + Element Plus。
  2. 后端：FastAPI + SQLAlchemy + Pydantic + Uvicorn。
  3. 数据层：MySQL 8（单库）。
  4. 任务调度：APScheduler（每日采集/训练）。
  5. 模型：基线模型（XGBoost/LightGBM 二选一，默认 LightGBM）+ 时间序列对比基线（Prophet 或 SARIMAX，默认 Prophet）。
  6. 可视化：前端 ECharts；分析报告图由后端生成 PNG/CSV。

  #### 3. 数据模型（核心表）

  1. fact_tourist_flow_daily：date, scenic_id, visitor_count, source, quality_flag。
  2. dim_weather_daily：date, temp_max, temp_min, rain, weather_type, wind_level。
  3. dim_calendar：date, is_holiday, holiday_name, is_weekend。
  4. dim_event：date, event_type, intensity, note。
  5. feature_dataset_daily：训练特征快照与版本。
  6. model_registry：模型版本、参数、指标、上线状态。
  7. pred_tourist_flow_daily：预测结果（date, horizon, y_pred, y_low, y_high, model_version）。
  8. sys_user/sys_role/audit_log：基础权限与审计。

  #### 4. 后端 API（公开接口约定）

  1. POST /api/v1/auth/login：登录换 token。
  2. POST /api/v1/data/import/{domain}：按域导入（flow/weather/calendar/event）。
  3. GET /api/v1/data/preview：预览筛选数据。
  4. POST /api/v1/features/build：构建指定时间段特征。
  5. POST /api/v1/models/train：触发训练（7/30 天模型）。
  6. GET /api/v1/models/metrics：查看 MAE/RMSE/MAPE/R2。
  7. POST /api/v1/models/predict：生成未来 7/30 天预测。
  8. GET /api/v1/predictions：查询预测与置信区间。
  9. GET /api/v1/analysis/correlation：影响因素相关性。
  10. GET /api/v1/dashboard/overview：看板聚合数据。
  11. GET /api/v1/export/report：导出报告（CSV/PNG）。

  #### 5. 算法与特征策略

  1. 特征：

  - 时间特征（周几、月份、季节、节假日窗口）。
  - 滞后特征（1/7/14/30 天）。
  - 滚动统计（7/14/30 天均值与波动）。
  - 气象特征与事件特征。

  2. 建模流程：

  - 时间切分训练/验证/测试（禁止随机打散）。
  - 多模型对比后固定主模型。
  - 预测输出点预测+区间预测。

  3. 解释性：

  - SHAP 或特征重要性条形图用于答辩解释“影响因素”。

  ### 四、测试与验收标准

  #### 1. 功能测试

  1. 导入模板校验、缺失值处理、重复数据拦截。
  2. 训练任务可执行、可追踪、可回放。
  3. 看板图表可联动筛选（时间/景区/因素）。

  #### 2. 性能测试

  1. 单次查询接口 P95 < 5 秒。
  2. 看板首屏加载 < 7 秒（样本量按一年日级数据+多景区）。
  3. 训练任务在可接受窗口内（例如 < 10 分钟）。

  #### 3. 模型验收

  3. 30 天预测：趋势方向正确率与误差并行评估。

  #### 4. 安全与稳定
  1. 账号鉴权、接口鉴权、基础审计日志可查。
  2. 数据库每日备份，恢复演练至少 1 次。
  3. 导入失败可回滚，不污染主表。

  ### 五、里程碑（与文档20周对齐，细化为可执行）

  1. 第1-2周：需求冻结、数据字典、原型图、接口草案。
  2. 第3-4周：环境搭建、数据库建表、鉴权框架。
  3. 第5-8周：数据导入与清洗链路、质量规则。
  4. 第9-12周：特征工程、模型训练评估、版本管理。
  5. 第13-16周：Vue 看板与分析页、预测页、报告导出。
  6. 第17-19周：联调、压测、缺陷修复、验收报告。
  7. 第20周：部署演示环境、操作手册、答辩材料。

  ### 六、关键假设与默认选择

  1. 已有“按天”游客量历史主数据，可覆盖至少 12 个月。
  2. 一期仅做单区域（荔波县）多景区扩展，不做跨地市泛化。
  3. 先采用 MySQL 单库，不启用 HDFS/NoSQL。
  4. 预测目标固定为未来 7 天与 30 天。
  5. 技术栈默认 FastAPI + Vue3，最符合文档且实现成本最低。
